Risk R-4: Scope creep delays v1.0 release. Mitigation: Strict phase gates and review points; defer non-critical features to Phase 4

## Risk Details

- **Likelihood:** Medium
- **Impact:** Medium
- **Mitigation:** Strict phase gates and review points; defer non-critical features to Phase 4

## Implementation

Phase gates have been implemented in the task management system to ensure proper review processes and prevent scope creep.

### Features Added

1. **New Status: `in-review`**
   - Tasks must pass through `in-review` before being marked `done`
   - Enforced at the API level in `internal/task/task.go`

2. **Phase Gate Validation**
   - `ValidatePhaseGate()` method prevents invalid status transitions
   - Valid flow: `todo` → `in-progress` → `in-review` → `done`
   - Rework flow: `in-review` → `in-progress` (send back for fixes)
   - Reset: any state → `todo` (explicit reset)
   - Block: any state → `blocked`

3. **Phase Tags**
   - Tasks can be tagged with `phase-N` (e.g., `phase-1`, `phase-2`)
   - `HasPhaseTag()` and `GetPhase()` methods for phase identification
   - Helps defer non-critical features to later phases

4. **CLI Commands**
   - `tssk phase check` - Verify all tasks follow phase gate rules
   - `tssk phase list` - List tasks grouped by phase

5. **CI Enforcement**
   - Phase gate check added to CI workflow
   - Runs on every PR and push to main
   - Warns about compliance violations

### Status Transition Diagram

```
todo → in-progress → in-review → done
  ↑         ↑           ↓
  └─────────┴───────────┘ (rework/reset)
  
any state → blocked
```

### Benefits

- Ensures code review before completion
- Provides visibility into task phases
- Prevents shortcutting the review process
- Enables phase-based release planning
